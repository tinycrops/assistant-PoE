# Applying Character-Centered Reasoning Research to Assistant-PoE

## A Research Report on the First Repo-Native Benchmark and Model Fit

**Date:** 2026-03-23  
**Repository:** `assistant-PoE`  
**Application focus:** converting existing character traces into a trainable benchmark  
**Research artifact location:** `research/character_continuity_benchmark/`

## Abstract

This report documents the first direct application of our structured-reasoning research program to `assistant-PoE`. Instead of starting with synthetic labels or writing new gameplay simulators, we asked a narrower and more repo-native question:

> Given the current traced state of a Path of Exile character, can a model identify the true immediate next state of that character from a mixed candidate set?

To answer that, we built a benchmark entirely from existing `assistant-PoE` artifacts, primarily `logs/stat_watch/*_history.jsonl`. We reconstructed cumulative character states, represented them as stat-and-gear vectors, and cast progression continuity as a ranking problem. We then trained a small neural ranker and compared it to a hand-designed heuristic baseline.

The resulting benchmark contains 20 reconstructed states across 3 characters, producing 12 training anchors and 5 held-out evaluation anchors. On this benchmark, the heuristic achieved `top1_accuracy = 0.80` and `MRR = 0.85`, while the learned ranker achieved `top1_accuracy = 1.00` and `MRR = 1.00` on the held-out set.

These results are small-data and preliminary, but they demonstrate real traction: the repository already contains enough structured temporal signal for a model to learn something meaningful about character continuity. This matters because `assistant-PoE` is not just a toolkit of scripts; it is already a character-centered intelligence system with persistent memory, grounded observations, and action-oriented outputs.

## Why This Domain Was a Strong Fit

The retrospective report in `docs/research/repo-retrospective-2026-03-22.md` argued that `assistant-PoE` has already become a character-centered intelligence system. That thesis is not metaphorical. It appears concretely in the implementation:

- `defaults.env` defines a live active character context.
- `poe_stat_watch.py` turns live account state into headless-PoB-backed mechanical state.
- `character_ledger.py` persists memory at the character level.
- `weighted_trade_search.py` gates decision-making on freshness of grounded state.
- `poe_market_pipeline.py` and Discord outputs externalize understanding as action prompts.

This architecture is exactly the kind of environment where structured reasoning research becomes valuable. The system already has:

- persistent entities,
- typed state transitions,
- grounded observations,
- action-oriented outputs,
- and durable historical traces.

That makes `assistant-PoE` a much better candidate for our work than a generic chat app or a purely aesthetic game companion.

## Research Question

The first applied research question was:

> Can a model learn character continuity directly from the traces already stored in this repository?

This question was chosen because it aligns with the repo’s deepest claim:

> A character is a living intelligence object, not a disposable current-state payload.

If that claim is true in operational terms, then continuity should be measurable and learnable from the trace data.

## Method

### Data source

Primary source:

- `logs/stat_watch/*_history.jsonl`

These files record chronological stat-change and equipment-change events generated by the headless PoB tracking flow.

### State reconstruction

Each history stream was converted into cumulative state nodes by:

1. Starting from zeroed tracked stats and empty gear occupancy.
2. Applying each `stat_changes` entry as a new “after” value.
3. Parsing `equipped_changes` into slot occupancy transitions.
4. Emitting a cumulative state after every history row.

### Feature space

Tracked signals included:

- 28 numeric stat dimensions, including:
  - `defence.life`
  - `defence.energy_shield`
  - `defence.mana`
  - `defence.fire_resist_percent`
  - `defence.cold_resist_percent`
  - `defence.lightning_resist_percent`
  - `offence.total_dps`
  - `offence.average_hit`
  - `offence.attack_or_cast_rate`
  - plus charge, mitigation, and attribute features
- 16 gear-slot occupancy features, including:
  - `Weapon`
  - `Offhand`
  - `Helm`
  - `BodyArmour`
  - `Gloves`
  - `Boots`
  - `Belt`
  - `Amulet`
  - `Ring`
  - `Ring2`
  - flask occupancy slots

### Task definition

For each anchor state:

- one candidate is the true next chronological state for that same character,
- the remaining candidates are distractors from:
  - non-adjacent states of the same character,
  - or states from other characters.

The model must rank the true next state highest.

### Metrics

- `top1_accuracy`
- `MRR`

## Baselines

### Heuristic baseline

The heuristic ranks candidates using:

- same-character preference,
- smaller cumulative stat delta from the anchor,
- smaller positive time delta.

This is not a random baseline. It is a strong hand-built continuity prior.

### Learned baseline

The learned model is a small MLP over pair features derived from:

- anchor stats,
- candidate stats,
- signed deltas,
- absolute deltas,
- slot occupancy transitions,
- same-character indicator,
- and time delta features.

## Interpretation

The result should be interpreted carefully.

### What it does show

- The repository contains enough structured temporal signal to support a trainable benchmark.
- Character continuity is not just a conceptual framing; it is measurable.
- A learned model can outperform a strong heuristic on held-out continuity ranking.
- The existing stat-watch traces are already rich enough to support experimental work.

### What it does not show

- It does not show broad Path of Exile competence.
- It does not show reliable action planning yet.
- It does not prove the model can recommend upgrades, trades, or gameplay decisions.
- It does not validate large-scale compositional reasoning in the full sense of our broader research program.

## Why This Result Matters for Assistant-PoE

The most important consequence is not the exact `1.00` number. It is the fact that the repo’s traces were structured enough to make this possible at all.

That matters because it means:

1. `assistant-PoE` already has benchmarkable internal state.
2. The repo can support research loops without requiring manual relabeling from scratch.
3. The character-centered thesis is operational enough to be tested directly.
4. We now have a path from “trace collection” to “trainable character intelligence.”

## Limitations

This experiment has several obvious limitations.

### 1. Small data

The benchmark is tiny:

- 20 states
- 12 train anchors
- 5 eval anchors

This is only enough for a traction probe.

### 2. Narrow eval slice

Most held-out examples come from the tail of `collabwitch_codex`, so this is not yet a robust cross-character generalization result.

### 3. Simplified task

Continuity ranking is easier than open-ended action generation. It tests whether the system can track progression structure, not whether it can yet produce fully correct next-step advice.

### 4. Candidate construction

Negatives are meaningful but still somewhat shallow. Future versions should include harder negatives:

- same-character near-neighbors,
- states with similar stats but different gearing semantics,
- and market-conditioned distractors.

## Novelty of the Benchmark

The benchmark is novel within this repository because it does not ask:

- “What is the current character state?”
- or “What advice should we give?”

Instead, it asks:

> Does the system understand how a character evolves through time well enough to recognize the true next state?

That is a more fundamental question. If a model cannot track continuity, it has no business making confident long-horizon recommendations.

## Next Steps

The most promising next experiments are:

### 1. Expand the dataset

Pull in more evidence from:

- archived stat-watch snapshots,
- character journals,
- market-sync events,
- build-intel artifacts,
- and future session runs.

### 2. Add a second benchmark

Two strong candidates:

- `state -> next action prompt`
- `state -> journal event type / milestone summary`

### 3. Harder negatives

Construct candidates that are:

- temporally close,
- same-character but wrong-step,
- or economically plausible but mechanically wrong.

### 4. Move from continuity to action

Once continuity tracking is stable, the next layer is:

- grounded trade recommendation ranking,
- upgrade prioritization,
- or memory-conditioned build-intel generation.

### 5. Scale when the task is stable

Kaggle GPUs/TPUs should be used after the benchmark contract is expanded and hardened, not before.

## Conclusion

This was the first serious application of our structured-reasoning work to `assistant-PoE`, and it succeeded in the most important way: it found real signal in the repository’s existing traces and converted that signal into a benchmark a model could learn.

The experiment is small, but it changes the status of the project. `assistant-PoE` is no longer just a repository with clever tools and nice framing. It is now demonstrably a source of trainable character-continuity data.

That is exactly the foothold we wanted.
